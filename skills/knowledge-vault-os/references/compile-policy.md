# Compile Policy

## Goal

Compilation turns evidence into reusable knowledge. The output should help a future person or agent make a better decision faster.

## Promotion Checklist

A claim may be compiled only when it is:

- Traceable to a source packet, path, link, or dated note.
- Reusable beyond the current task.
- Safe at its written sensitivity level.
- Targeted to one clear page type.
- Clear about whether it adds, replaces, or conflicts with existing knowledge.

## Page Types

| Folder | Use For |
|---|---|
| `20-projects/` | Project state, scope, artifacts, next actions, blockers |
| `30-decisions/` | Confirmed choices, boundaries, tradeoffs |
| `40-methods/` | Repeatable frameworks, workflows, content methods |
| `50-context/` | Stable personal, company, domain, or operating context |
| `60-ai-skills/` | Skills, agent rules, prompts, automation contracts |
| `80-agent-briefs/` | Condensed context agents must read first |

## Update Strategy

Prefer updating existing pages. Create a new page only when:

- No existing page owns the theme.
- The topic will recur.
- The title will still make sense in three months.

## Project Status Card

Project pages should expose these fields when useful:

- `project_stage`
- `latest_progress`
- `key_judgment`
- `next_action`
- `blockers`

## Sensitive Evidence

For L4 sources, compile only project-level, process-level, method-level, or management-level conclusions. Do not copy raw chat lines, names, salary details, candidate details, family details, customer details, or private one-on-one content into topic pages unless the user explicitly authorizes it and the note sensitivity matches.
