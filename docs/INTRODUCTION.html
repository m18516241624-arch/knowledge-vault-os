Codex 可安装的本地知识库操作系统

Knowledge Vault OS 把资料变成可调用的长期知识

它不是 Obsidian 模板，也不是把文件丢给 RAG 的问答库。它是一套从感知、证据、编译到 Agent 调用的工作流，让 Codex 能持续维护你的 Markdown 主库。

先从最基础的问题说起

很多人已经有 Obsidian、文件夹、AI 聊天记录和各种收藏，但它们通常只是堆在一起。Knowledge Vault OS 做的事，是给这些信息一条可控的晋升路径。

1

为什么不是直接问 AI？

直接问 AI 适合临时答案，但不适合沉淀长期知识。项目判断、个人方法、组织经验和决策边界需要被写回稳定位置，下次才能继续使用。

2

为什么不是普通笔记模板？

模板只规定文件夹怎么摆。这个项目还规定来源怎么进来、证据怎么登记、什么时候能晋升、Codex 做事前应该读哪些入口。

3

为什么强调本地优先？

个人资料、聊天记录、公司上下文和客户信息都有敏感边界。Markdown 主库在本地，外部工具只做辅助，不替代事实源。

三个核心概念

理解这三个词，就能理解整套系统为什么能从“资料收集”走到“智能体调用”。

A

Source Packet

每个来源先变成来源包，里面记录路径、时间窗口、敏感等级、候选结论和下一步。它是证据，不是最终知识。

B

Topic Page

稳定、可追溯、可复用的结论，才会进入项目页、决策页、方法页或上下文页。这里是长期可读的知识主干。

C

Agent Brief

给 Codex 和其他 agent 读取的压缩入口。它不保存所有原文，只保存未来做事必须先知道的上下文。

从感知到调用

这条链路保证新信息不会直接变成“真相”，而是先留下证据，再经过判断和编译。

Step 01

感知

扫描本地项目、Markdown 笔记、导出文件或其他来源。

scripts/daily-capture.mjs
Step 02

来源登记

明确来源是谁、什么时候产生、敏感等级是什么、是否允许自动处理。

_maintenance/source-registry.md
Step 03

证据入账

把原始线索写入 inbox、source packet 和 capture ledger。

00-inbox/source-packets/
Step 04

稳定性判断

判断哪些内容只是临时记录，哪些内容可以成为长期复用的结论。

decision: review / compile
Step 05

主题页编译

把明确可复用的结论写入项目、决策、方法、上下文或 AI Skill 页面。

scripts/compile-inbox.mjs
Step 06

Agent 调用

Codex 开始任务前读取全局索引、当前重点和任务相关主题页。

80-agent-briefs/

五层架构

它不是把所有东西塞进一个库里，而是把信息按职责分层。每一层只做自己该做的事。

1

来源治理层

定义来源等级、敏感边界、入口路径和能否自动编译。

2

证据层

承接原始资料和来源包，保留事实线索，避免污染长期主题页。

3

知识编译层

只接收稳定、可追溯、可复用、已脱敏的结论。

4

Agent 调用层

用全局索引和 Agent Brief 让 Codex 快速读懂当前上下文。

5

本地可视化层

Dashboard Lite 只读本地 vault，用于查看分布、证据队列和健康问题。

Git

版本与发布

Git 负责回滚、迁移和公开发布；发布包不包含用户真实私密资料。

一天里怎么用

新用户可以先不用理解所有文件夹。按下面这条日常路径，就能让系统运转起来。

git clone https://github.com/m18516241624-arch/knowledge-vault-os.git
cd knowledge-vault-os
npm install
npm run validate

node scripts/init-vault.mjs ~/Documents/MyKnowledgeVault
node scripts/daily-capture.mjs ~/Documents/MyKnowledgeVault ~/Documents/MyProject --target-date YYYY-MM-DD
node scripts/compile-inbox.mjs ~/Documents/MyKnowledgeVault --dry-run
node scripts/build-dashboard-lite.mjs ~/Documents/MyKnowledgeVault
先初始化 vault

生成一套 Markdown 目录、模板和 Agent 入口文件，Obsidian 可以直接打开。

再捕获当天来源

daily-capture 会记录变更文件，但不会擅自总结成长期知识。

人工或 Codex 做晋升判断

只有当 source packet 写明 decision: compile、稳定结论和目标页面时，才进入编译。

最后让 Codex 调用

做项目、写文章、整理方法或处理需求前，让 Codex 先读 Agent Brief 和相关主题页。

包里具体有什么

v0.2.0 已经不是一个空壳，而是一套可以安装、验证、打包和重复运行的本地工作流。

Codex Plugin

.codex-plugin/plugin.json 描述插件入口和安装信息。

Main Skill

skills/knowledge-vault-os/ 规定 Codex 如何初始化、接入、编译和调用。

Vault Template

assets/vault-template/ 提供公开安全的 Markdown 主库结构。

Daily Capture

扫描本地来源目录，写 source packet、ledger 和 daily log。

Compile Inbox

只编译明确标记的来源包，重复运行不会重复追加。

Dashboard Lite

静态页面读取本地 JSON，展示分布、项目、证据队列和健康问题。

Validation

npm run validate 覆盖 manifest、Skill、demo vault 和隐私扫描。

Release Build

npm run package:plugin 生成本地发布归档，便于后续分发和校验。

适合谁，不适合谁

它适合把 AI 当长期协作者的人，而不是只想做一次性资料问答的人。

适合

  • 长期使用 Codex，希望它记住项目上下文的人。
  • 用 Obsidian / Markdown 做个人知识主库的人。
  • 希望把自己的知识整理方法打包给别人安装的人。
  • 重视隐私边界，不想把原始聊天和公司资料直接外传的人。

暂不适合

  • 只想要一个云端问答机器人。
  • 希望自动抓取所有账号、聊天和附件。
  • 希望开箱即用团队权限、云同步和向量数据库。
  • 不愿意维护 Markdown 主库和晋升规则。

先把主链跑通，再谈更复杂的自动化

v0.2.0 的重点是让“本地优先、证据先行、谨慎编译、Agent 调用”成为可安装、可验证、可发布的基础能力。
